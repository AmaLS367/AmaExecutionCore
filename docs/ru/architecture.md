# Архитектура

AmaExecutionCore организован вокруг конвейера (pipeline), описанного в `AGENTS.md`:

`market_data -> strategy_engine -> risk_manager -> safety_guard -> order_executor -> exchange_sync & trade_journal`

Этот полный конвейер в настоящее время частично реализован через два конкретных потока ввода.

## Текущие потоки выполнения во время работы (Runtime Entry Flows)

### Поток выполнения API

`signal_execution -> order_executor -> exchange_sync -> trade_journal`

Этот путь используется, когда внешний вызывающий компонент уже имеет полностью сформированную торговую идею и отправляет ее на `POST /signals/execute`.

### Теневой поток конвейера (Shadow pipeline flow)

`market_data -> strategy_engine -> risk_manager -> safety_guard -> order_executor -> trade_journal`

Текущий теневой конвейер (shadow pipeline) — это точка входа по требованию, реализованная с помощью `backend/backtest/shadow_runner.py`:

1. `market_data`
   `BybitSpotSnapshotProvider` получает недавние спотовые свечи через Bybit REST и нормализует их в минимальный `MarketSnapshot`, состоящий из упорядоченных свечей с `high`, `low` и `close`.
2. `strategy_engine`
   `StrategyExecutionService` получает явные `symbol` и `interval`, нормализует их на границе сервиса, запрашивает окно свечей, необходимое для стратегии, и запускает стратегию на нормализованном снимке.
3. `risk_manager -> safety_guard -> order_executor`
   `OrderExecutor` по-прежнему владеет проверками перед выполнением: расчет размера позиции, валидация минимального RRR (Risk-to-Reward Ratio), проверка лимита открытых позиций, проверка общего воздействия (exposure), проверка кнопки экстренной остановки (kill switch) и автоматического выключателя (circuit-breaker).
4. `trade_journal`
   В режиме `shadow` сделка сохраняется с записью о выполнении, но размещение ордера останавливается до отправки любого ордера через REST на биржу.

Этот теневой поток не является механизмом исторического воспроизведения, а также планировщиком или демоном. Он запускается один раз для каждого явного вызова с предоставленными `symbol` и `interval`.

`position_manager` остается смежным с этими потоками ввода. Он не открывает позиции; он управляет запросами оператора/демо на закрытие для сделок, которые уже считаются открытыми на стороне биржи.

## Ответственность (Ownership)

- `market_data`: отвечает за получение и нормализацию снимка рынка. Текущая реализация — это поставщик снимков на базе Bybit Spot REST для первой стратегии.
- `strategy_engine`: производит `StrategySignal` из нормализованных снимков и никогда не обращается напрямую к бирже или сохранению в БД.
- `signal_execution`: сохраняет `Signal`, вызывает `OrderExecutor` и возвращает текущий сохраненный снимок сделки.
- `risk_manager`: в настоящее время вносит математику рисков и валидацию на стороне выполнения, используемую `OrderExecutor`.
- `safety_guard`: обеспечивает работу постоянной кнопки экстренной остановки, ежедневных/еженедельных пауз и окон охлаждения.
- `order_executor`: выполняет проверки рисков, проверки безопасности, проверки воздействия, идемпотентность, а также отправку ордера или сохранение в теневом режиме.
- `exchange_sync`: отвечает за сверку ордеров биржи после того, как ордер получит идентификатор биржи. Он потребляет приватные события Bybit WebSocket и периодически перепроверяет сохраненные ID ссылок на ордера входа/закрытия через REST.
- `position_manager`: отвечает за определение во время выполнения того, "достаточно ли открыта сделка для управления", и отвечает за отправку запроса на закрытие плюс повторную попытку закрытия после `POSITION_CLOSE_FAILED`.
- `trade_journal`: хранит `Signal`, `Trade`, ежедневную статистику, системные события и состояние безопасности.
- `backtest`: в настоящее время содержит многократно используемую точку входа `ShadowRunner` и отдельный помощник `DemoRunner` для тестовой сети (testnet).

## Контракт снимка стратегии

Первая стратегия — `EMACrossoverStrategy`.

Ее текущие ожидания намеренно минимальны:

- `symbol`
- `interval`
- упорядоченное окно свечей с `high`, `low` и `close`
- как минимум `slow + 1` свечей, чтобы пересечение EMA могло сравнить две последние точки EMA

Никакой абстракции исторического воспроизведения, потока тикеров или более широкой оркестрации рыночной вселенной (market-universe) пока не добавлено.

### Истина открытой позиции

Надежной записью всегда является сохраненная строка `Trade` в базе данных.

`position_manager` владеет интерпретацией того, какие статусы по-прежнему представляют собой активное воздействие на биржу для потоков, ориентированных на оператора:

- `POSITION_OPEN`
- `ORDER_PARTIALLY_FILLED`
- `POSITION_CLOSE_PENDING`
- `POSITION_CLOSE_FAILED`

Это право собственности используется `/positions/open` и правилами повторной попытки закрытия. `exchange_sync` обновляет эти статусы на основе данных со стороны биржи, но он не решает, какие из них должны оставаться видимыми/управляемыми как открытые позиции.

### Ответственность за восстановление

- Сверка входных ордеров принадлежит `exchange_sync`.
  Он перепроверяет сделки в статусах `ORDER_SUBMITTED` и `ORDER_PENDING_UNKNOWN`, используя сохраненный `order_link_id`.
- Сверка ордеров на закрытие принадлежит `exchange_sync`.
  Он перепроверяет сделки в статусе `POSITION_CLOSE_PENDING`, используя сохраненный `close_order_link_id`.
- Восстановление после сбоя закрытия принадлежит `position_manager`.
  Когда ордер на закрытие отклоняется или отменяется, `exchange_sync` помечает сделку как `POSITION_CLOSE_FAILED`; последующий ручной/демо-запрос на закрытие от `position_manager` создает заменяющий ордер на закрытие.
- Ни `exchange_sync`, ни рабочий процесс сверки (reconciliation worker) не создают новых сделок.
  Восстановление изменяет только существующую строку `Trade`, которая уже содержит сохраненные идентификаторы биржи.

## Постоянная модель безопасности

`safety_state` — это единственный источник истины для:

- `kill_switch_active`
- `pause_reason`
- `cooldown_until`
- `manual_reset_required`
- `last_triggered_at`

Объект `KillSwitch` в памяти отражает сохраненное состояние, но не заменяет его.

## Заметки о жизненном цикле

- Теневой режим (Shadow mode) сохраняет запись `Trade`, но не вызывает конечные точки размещения ордеров на бирже.
- Текущий теневой путь, следовательно, останавливается перед `exchange_sync`; на стороне биржи нет ордера для сверки.
- Тайм-ауты открытых ордеров переводят сделки в статус `ORDER_PENDING_UNKNOWN`.
- Ордера на вход и закрытие используют разные ID ссылок.
- Рабочий процесс сверки (reconciliation worker) выполняется в течение всего времени жизни приложения для нетеневых режимов.
- Если событие WebSocket пропущено, задержано или поступает не по порядку, рабочий процесс перепроверяет сохраненные нетерминальные сделки через REST и сводит существующее состояние сделки (converges trade state), когда состояние на стороне биржи становится известным.
- Если REST все еще не может разрешить состояние на стороне биржи, сделка остается в своем текущем нетерминальном статусе и имеет право на следующий проход сверки, в том числе после перезапуска процесса.
- Выполнение ордеров на закрытие переводит сделки из статуса `POSITION_CLOSED` в `PNL_RECORDED`.
- `DailyStat` обновляется из реализованных событий закрытия, а не только из проверок перед сделкой.
