# Руководство по бэктестированию и воспроизведению

Директория `backend/backtest/` теперь содержит три различных вспомогательных модуля (helpers) времени выполнения с разными целями. Они не взаимозаменяемы.

## Историческое воспроизведение (Historical Replay)

Историческое воспроизведение реализовано через `backend.backtest.HistoricalReplayRunner`.

Контракт входа:

- `HistoricalReplayRequest.symbol`
- `HistoricalReplayRequest.interval`
- ровно одно из следующего:
  - `candles`: историческая последовательность свечей
  - `snapshots`: явная историческая последовательность снимков
- необязательный `start_step`
- необязательный `end_step`

Семантика шага:

- `end_step` является исключающим
- при воспроизведении `candles`, runner строит скользящие снимки (rolling snapshots), используя `required_candle_count` активной стратегии
- при воспроизведении `snapshots`, runner повторно использует предоставленные снимки "как есть"

Форма результата воспроизведения:

- `request`: нормализованный запрос воспроизведения
- `steps`: упорядоченные шаги воспроизведения
- `report`: машиночитаемые метрики, полученные из известных результатов выполнения

Каждый шаг воспроизведения содержит:

- `step_index`
- `snapshot`
- `signal`
- `execution`

Это позволяет использовать вывод воспроизведения в тестах, скриптах или последующей автоматизации без добавления UI/рендеринга отчетов.

## Метрики и отчетность

`HistoricalReplayResult.report` в настоящее время предоставляет:

- `metrics.closed_trades`
- `metrics.winning_trades`
- `metrics.losing_trades`
- `metrics.expectancy`
- `metrics.win_rate`
- `metrics.profit_factor`
- `metrics.max_drawdown`
- `slippage.count`
- `slippage.average`
- `slippage.minimum`
- `slippage.maximum`

Важные ограничения:

- метрики вычисляются только из результатов выполнения, которые фактически раскрывают `realized_pnl` (реализованный PnL)
- сводка по проскальзыванию (slippage summary) вычисляется только тогда, когда результаты выполнения раскрывают `slippage`
- если выполнение воспроизведения не предоставляет эти значения, отчет оставляет затронутые поля как `None`
- никаких синтетических исполнений (fills), предполагаемого PnL или угаданной аналитики не добавляется

## Использование стратегии

Воспроизведение работает с любой стратегией, которая следует существующему контракту `BaseStrategy`.

Для запусков с несколькими стратегиями, `backend.strategy_engine.StrategyOrchestrator` предоставляет минимальный уровень упорядоченной композиции:

- он оценивает стратегии по порядку
- первый сигнал, отличный от `None`, выигрывает
- требования к снимкам расширяются до наибольшего `required_candle_count` в наборе

Это намеренно узко. Это не общий фреймворк плагинов и не система планирования.

## Replay vs Shadow vs Demo

Используйте правильный инструмент для правильной задачи:

- `HistoricalReplayRunner`: воспроизведение исторической последовательности для оценки в стиле бэктеста
- `ShadowRunner`: одноразовый путь выполнения (runtime path) от снимка -> стратегии -> сервиса исполнения, обычно с `TRADING_MODE=shadow`
- `DemoRunner`: опциональный помощник Bybit testnet для проверки работы с внешними сервисами

Текущие границы:

- воспроизведение не зависит от подключения к live/demo
- теневой режим (shadow mode) по-прежнему является путем проверки ядра исполнения, а не историческим симулятором
- демо-режим (demo mode) по-прежнему является помощником для testnet, а не обобщенным средством запуска бэктестов

## Simulation Execution

`backend.backtest.SimulationExecutionService` теперь используется как paper execution helper для replay-сценариев, которым нужен доступ к будущим свечам после текущего шага.

Поведение:

- получает уже собранный `ExecuteSignalRequest`
- получает срез будущих свечей после текущего replay-step
- проверяет TP, SL или timeout по этим свечам
- возвращает машиночитаемый результат с `realized_pnl`, `slippage`, `exit_reason` и `hold_candles`

Этот helper используется в:

- `scripts/run_backtest.py`
- `scripts/validate_strategy.py`

Текущие validation thresholds:

- `win_rate > 0.55`
- `expectancy > 0`
- `max_drawdown < 10 * avg_risk_amount`
- `closed_trades >= 30`

Скрипт валидации применяет поправку на round-trip fee до вынесения финального pass/fail решения.

## Текущие ограничения

Текущая реализация воспроизведения намеренно узкая:

- нет планировщика пакетных заданий (batch job scheduler)
- нет HTML или дашборд-отчетности
- нет автоматической загрузки исторических данных
- нет портфельной аналитики за пределами представленных агрегированных метрик
- нет претензий на то, что поведение при воспроизведении идеально совпадает с исполнением на бирже

Используйте воспроизведение, чтобы оценить поведение стратегии на предоставленной исторической последовательности, а не для вывода гарантий исполнения производственного уровня.
