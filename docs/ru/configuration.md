# Руководство по конфигурации

В этом документе объясняются все переменные окружения, которые находятся в файлах `.env.example` / `.env`. Эти настройки контролируют основное поведение бота, параметры подключения, границы рисков и уровни безопасности.

---

## ⚙️ Общие настройки

| Переменная | Тип | По умолчанию | Описание |
|---|---|---|---|
| `ENVIRONMENT` | `string` | `development` | Определяет среду выполнения. Может быть `development` (разработка) или `production` (продакшен). |
| `DEBUG` | `bool` | `True` | Включает режим отладки для FastAPI и более подробные трассировки стека. |
| `LOG_LEVEL` | `string` | `DEBUG` | Стандартный уровень логирования Python: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`. |

---

## 🏦 Учетные данные API Bybit

| Переменная | Тип | По умолчанию | Описание |
|---|---|---|---|
| `BYBIT_TESTNET` | `bool` | `True` | Принудительно подключает бота к Bybit Testnet (`True`) или к основной сети Bybit Mainnet (`False`). |
| `BYBIT_API_KEY` | `string` | `''` | Ваш публичный ключ API Bybit. Обязателен, если не включен режим Shadow. |
| `BYBIT_API_SECRET` | `string` | `''` | Ваш секретный ключ API Bybit. Держите его в безопасности и никогда не коммитьте его в репозиторий! |

## 📈 Торговый движок (Trading Engine)

| Переменная | Тип | Ожидаемые значения | Описание |
|---|---|---|---|
| `TRADING_MODE` | `string` | `shadow`, `demo`, `real` | Управляет потоком выполнения. <br>• **shadow**: Вычисляет все, но *не размещает* сделки на бирже.<br>• **demo**: Выполняется на Bybit demo/testnet.<br>• **real**: Разрешено ставить реальные средства под угрозу риска. |
| `ORDER_MODE` | `string` | `maker_only`, `maker_preferred`, `taker_allowed` | Влияет на типы запросов ордеров.<br>• **maker_only**: Принудительно "Post-Only". Ордер отклоняется, если он исполняется немедленно.<br>• **maker_preferred**: Сначала пробует "Post-Only", в качестве запасного варианта использует рыночный (market) ордер.<br>• **taker_allowed**: Разрешает немедленное исполнение рыночного ордера. |
| `SHADOW_EQUITY` | `float` | `10000.0` | Смоделированный капитал счета (equity), используемый в теневом режиме для расчета размера позиции. |
| `USE_TRAILING_STOP` | `bool` | `False` | Определяет, следует ли отправлять на биржу плавающие защитные стопы (trailing safety stops). Намеренно отключено для MVP. |
| `DEMO_CLOSE_TTL_SECONDS` | `int` | `30` | Как долго модуль запуска демо (demo runner) ждет после того, как вход достигает статуса `POSITION_OPEN`, прежде чем он отправит рыночный ордер на закрытие. |
| `DEMO_POLL_INTERVAL_SECONDS` | `float` | `1.0` | Интервал опроса, используемый модулем запуска демо во время ожидания переходов состояний БД, управляемых событиями WebSocket. |
| `DEMO_TESTNET_SYMBOL` | `string` | `''` | Опциональный символ для сквозного потока live testnet. |
| `DEMO_TESTNET_ENTRY` | `float` | `0.0` | Лимитная цена входа, используемая потоком testnet e2e. |
| `DEMO_TESTNET_STOP` | `float` | `0.0` | Цена стоп-лосса, используемая потоком testnet e2e. |
| `DEMO_TESTNET_TARGET` | `float` | `0.0` | Целевая цена (take profit), используемая потоком testnet e2e. |
| `SPOT_EXIT_MONITOR_INTERVAL_SECONDS` | `float` | `5.0` | Интервал fallback-монитора для `spot market` сделок без защитных ордеров, сохранённых в БД. |

## 🔁 Автономный signal loop

| Переменная | Тип | По умолчанию | Описание |
|---|---|---|---|
| `SIGNAL_LOOP_ENABLED` | `bool` | `False` | Включает автономный цикл стратегий на основе REST-опроса для day-trading интервалов. |
| `SIGNAL_LOOP_SYMBOLS` | `list[str]` | `[]` | Список символов через запятую, например `BTCUSDT,ETHUSDT`. |
| `SIGNAL_LOOP_STRATEGY` | `string` | `ema_crossover` | Day-trading стратегия для signal loop. Поддерживаемые значения: `ema_crossover`. |
| `SIGNAL_LOOP_INTERVAL` | `string` | `15` | Интервал Bybit kline, используемый циклом сигналов. |
| `SIGNAL_LOOP_COOLDOWN_SECONDS` | `int` | `300` | Пауза после входа по каждому символу. |
| `SIGNAL_LOOP_MAX_SYMBOLS_CONCURRENT` | `int` | `5` | Ограничение параллельных оценок стратегии по символам. |

Рекомендуемые day-trading значения:

- `SIGNAL_LOOP_STRATEGY=ema_crossover`
- `SIGNAL_LOOP_INTERVAL=15`
- `RISK_PER_TRADE_PCT=0.005`
- `MIN_RRR=1.5`
- `MAX_OPEN_POSITIONS=2`
- `MAX_TOTAL_RISK_EXPOSURE_PCT=0.03`
- `MAX_TRADES_PER_DAY=10`
- `MAX_DAILY_LOSS_PCT=0.02`
- `HARD_PAUSE_CONSECUTIVE_LOSSES=4`
- `COOLDOWN_HOURS=2`

|---|---|---|---|

| `SCALPING_COOLDOWN_SECONDS` | `int` | `120` | Пауза после входа по каждому символу в WebSocket scalping runner. |

## 💰 Управление рисками

| Переменная | Тип | По умолчанию | Описание |
|---|---|---|---|
| `RISK_PER_TRADE_PCT` | `float` | `0.01` (1%) | Процент от общего активного капитала, которым вы готовы рискнуть в одной сделке (расстояние от Входа до Стоп-лосса). |
| `MIN_RRR` | `float` | `2.0` | Минимально допустимое соотношение риска к прибыли (Risk-to-Reward ratio). Сигналы, предполагающие RRR ниже этого значения, будут отклонены автоматически. |
| `MAX_OPEN_POSITIONS` | `int` | `1` | Строгое ограничение на одновременные открытые позиции, чтобы избежать чрезмерного риска. |
| `MAX_TOTAL_RISK_EXPOSURE_PCT` | `float` | `0.03` (3%) | Абсолютная максимальная сумма всех плавающих рисков (комбинированный `RISK_PER_TRADE_PCT`), допустимая одновременно. |
| `MAX_TRADES_PER_DAY` | `int` | `10` | Жёсткий дневной лимит новых входов, применяемый до отправки ордера. Защищает автономные раннеры от runaway-сигналов. |

Рекомендуемые scalping overrides:

- `ORDER_MODE=maker_preferred`
- `RISK_PER_TRADE_PCT=0.003`
- `MIN_RRR=1.5`
- `MAX_OPEN_POSITIONS=2`
- `MAX_TOTAL_RISK_EXPOSURE_PCT=0.01`
- `MAX_TRADES_PER_DAY=30`
- `MAX_DAILY_LOSS_PCT=0.02`
- `MAX_CONSECUTIVE_LOSSES=4`
- `HARD_PAUSE_CONSECUTIVE_LOSSES=6`
- `COOLDOWN_HOURS=1`

## 🛡️ Служба безопасности (Safety Guard / Circuit Breaker)

Эти настройки критически важны. Они определяют, когда бот должен принудительно прекратить прием новых сигналов для предотвращения уничтожения капитала.

| Переменная | Тип | По умолчанию | Описание |
|---|---|---|---|
| `MAX_DAILY_LOSS_PCT` | `float` | `0.03` (3%) | Если ежедневный реализованный PnL падает ниже этого порога, все новые сигналы блокируются. |
| `MAX_WEEKLY_LOSS_PCT` | `float` | `0.05` (5%) | Жесткий еженедельный автоматический выключатель. Требует ручного вмешательства/сброса в случае нарушения. |
| `MAX_CONSECUTIVE_LOSSES` | `int` | `3` | Порог периода охлаждения. Как только сегодняшний счетчик последовательных убыточных сделок достигает этого значения, новые входы приостанавливаются до истечения окна охлаждения (cooldown) или пока оператор не сбросит состояние безопасности. |
| `HARD_PAUSE_CONSECUTIVE_LOSSES` | `int` | `5` | Порог жесткой полосы убытков. Как только сегодняшний счетчик последовательных убыточных сделок достигает этого значения, бот переходит в режим паузы, требующий ручного сброса. |
| `COOLDOWN_HOURS` | `int` | `4` | Как долго пауза охлаждения остается активной после достижения порога охлаждения. |

## Семантика состояния безопасности

- `kill_switch_active=true` блокирует все новые входы, пока `POST /safety/reset` не очистит его.
- `pause_reason=daily_loss` или `pause_reason=weekly_loss` также требуют `POST /safety/reset`.
- `pause_reason=hard_loss_streak` требует `POST /safety/reset`.
- `pause_reason=cooldown` очищается автоматически после `cooldown_until`, и во время выполнения также очищается активная полоса последовательных убытков, когда происходит этот автоматический сброс.

## Поведение сброса

- `POST /safety/reset` всегда очищает `kill_switch_active`, `pause_reason`, `cooldown_until` и `manual_reset_required`.
- Сброс паузы `cooldown` или `hard_loss_streak` также очищает текущий отслеживаемый ежедневный счетчик последовательных убытков.
- Сброс `daily_loss` или `weekly_loss` не стирает лежащую в основе статистику реализованных убытков. Если тот же ежедневный или еженедельный порог все еще нарушен, следующая проверка безопасности немедленно снова приостановит торговлю.
- Кнопка экстренной остановки (kill switch) никогда не закрывает позиции автоматически. Она только блокирует новые входы и отменяет отложенные ордера на вход в режимах `demo` / `real`.

## Заметки о Demo/Testnet

- Стандартный быстрый набор тестов не обращается к Bybit.
- `pytest -m testnet` активируется по желанию и требует действительных учетных данных демо-версии Bybit плюс поля цен `DEMO_TESTNET_*`.
- Рыночные ордера на покупку (Market buy orders) стандартизированы по базовому количеству (base quantity) через `marketUnit="baseCoin"`.
- `SCALPING_SYMBOLS` и `SIGNAL_LOOP_SYMBOLS` не должны пересекаться. Если пересечение есть, приложение аварийно завершит startup lifecycle.
